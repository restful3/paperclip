## F1

**위치**  
`docs/report/03-runtime-execution.md:17`

**현재 본문 인용**  
> t₄에 토큰 사용량이 cost event로 들어와 회사·에이전트·이슈 차원으로 롤업되고, 이 변화는 회사 단위 WebSocket 라이브 채널(`activity.logged` · `heartbeat.run.*`)로 보드 UI에 즉시 반영된다.

**문제 카테고리**  
정합성

**제안 수정안**  
> t₄에 토큰 사용량이 cost event로 기록되고, 회사·에이전트의 월간 spend가 갱신된다. 이슈·프로젝트·run은 cost event의 귀속 축으로 남는다. 관련 실행 상태와 예산 임계값 변화는 각각 `heartbeat.run.*` 또는 `activity.logged` 라이브 이벤트로 보드 UI에 반영된다.

**근거**  
`heartbeat.ts`는 usage/cost를 `agent_runtime_state`에 누적한 뒤 cost event를 생성하며, 이때 `issueId`, `projectId`, `heartbeatRunId`를 귀속 필드로 넣는다(`server/src/services/heartbeat.ts:6760-6804`). `costService.createEvent()`는 cost event 삽입 후 에이전트 월간 spend와 회사 월간 spend만 갱신한다(`server/src/services/costs.ts:66-99`). `cost_events` 스키마에서 `issue_id`, `project_id`, `heartbeat_run_id`는 참조/귀속 필드다(`packages/db/src/schema/cost_events.ts:13-18`). `activity.logged`는 `logActivity()`가 발행하는 이벤트이며(`server/src/services/activity-log.ts:65-98`), budget threshold crossing 때 activity가 기록된다(`server/src/services/budgets.ts:671-710`). 따라서 "회사·에이전트·이슈 차원으로 롤업" 및 cost event 자체가 곧바로 `activity.logged`로 반영된다는 표현은 구현보다 넓다.

## F6

**위치**  
`docs/report/03-runtime-execution.md:96`

**현재 본문 인용**  
> "Suspicious silence creates a medium-priority review issue. Critical silence raises that review issue to high priority and blocks the source issue on the explicit evaluation task without cancelling the active process." — `doc/execution-semantics.md` §10

**문제 카테고리**  
정합성

**제안 수정안**  
> "Suspicious silence creates a medium-priority watchdog recovery action for the selected recovery owner. Critical silence raises that recovery action to high priority and, when issue-backed evaluation is needed for correctness, blocks the source issue on the explicit evaluation task without cancelling the active process." — `doc/execution-semantics.md:366`

또는 직인용을 피하려면:

> Suspicious silence는 selected recovery owner에게 medium-priority watchdog recovery action을 만들고, critical silence는 이를 high priority로 올린다. correctness를 위해 issue-backed evaluation이 필요할 때만 원본 이슈를 explicit evaluation task에 block한다(`doc/execution-semantics.md:357-376`).

**근거**  
현재 원문은 "review issue"가 아니라 "watchdog recovery action"을 기본 표현으로 사용한다(`doc/execution-semantics.md:357-366`). Issue-backed implementation은 별도 조건부 형태로 설명된다(`doc/execution-semantics.md:362`, `doc/execution-semantics.md:366`). 따라서 현 직인용은 실제 문장과 불일치하고, line citation도 빠져 있다.

**문제 카테고리**  
정합성

**제안 수정안**  
> API 서버가 부팅되면 heartbeat startup reconciliation이 결정론적 순서로 돈다(`server/src/index.ts:723-760`). periodic recovery loop도 같은 핵심 순서를 반복한다(`server/src/index.ts:789-826`). 코드 2가 그 순서다.

**근거**  
`server/src/index.ts:676-681`은 runtime API candidate URL 구성이고, `693-717`은 persisted runtime services 및 cloud upstream runs의 별도 startup reconciliation이다. heartbeat startup recovery의 7단계 흐름은 `reapOrphanedRuns()`부터 `reconcileProductivityReviews()`까지 `server/src/index.ts:723-760`에 있다. periodic recovery loop는 `server/src/index.ts:789-826`에 별도로 있다. 현재 범위 `676-780`은 startup heartbeat 7단계 밖의 설정/별도 reconciliation을 포함하고, periodic recovery의 핵심 뒷부분은 포함하지 않는다.
