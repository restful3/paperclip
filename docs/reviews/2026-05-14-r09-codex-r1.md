외부 SSE/heartbeat 설명의 큰 방향과 Paperclip 의 WebSocket/SSE 브리지 존재 여부는 맞습니다. 다만 몇몇 직접 인용·운영 세부와 §5의 Paperclip 적용 문장이 현재 코드의 실행 모델을 부정확하게 섞고 있어 수정이 필요합니다.

### 1. SSE 재연결을 지수 백오프로 단정

**위치**: L13

**현재**: 연결이 끊기면 user agent 가 지수 백오프로 재접속을 시도하며, `Last-Event-ID` 헤더로 마지막 수신 이벤트 ID 를 보내 서버가 그 지점부터 재개할 수 있게 한다.

**문제**: [ACCEPT] 사실 정확성 위반. WHATWG 는 `Last-Event-ID` 헤더와 `retry` 필드가 event stream 의 reconnection time 을 정한다고 설명하지만, "exponential backoff" 를 표준 동작으로 규정하지 않는다(https://html.spec.whatwg.org/multipage/server-sent-events.html §9.2.4, §9.2.6). MDN 도 연결 종료 시 기본 재시작과 `retry` 필드의 millisecond delay 를 설명할 뿐 지수 백오프를 말하지 않는다(https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events).

**제안**: "연결이 끊기면 user agent 가 내장 재연결 절차를 수행하며, 서버는 `retry:` 필드로 재연결 지연(ms)을 힌트로 줄 수 있다"처럼 지수 백오프 단정을 제거한다.

### 2. Anthropic 직접 인용 불일치

**위치**: L41

**현재**: 에이전트가 "operate autonomously over multiple turns" 하면서 "plan and operate independently, potentially returning to the human for further information or judgement" 한다고 본다.

**문제**: [ACCEPT] 사실 정확성 위반. Anthropic 원문에는 "operate autonomously over multiple turns" 라는 직접 문장이 없다. 원문은 일부 고객 정의로 "operate independently over extended periods" 를 말하고, agents 섹션에서는 "potentially operate for many turns" 와 "plan and operate independently..." 를 별도로 말한다(https://www.anthropic.com/engineering/building-effective-agents).

**제안**: 첫 따옴표는 paraphrase 로 바꾸거나, 원문에 맞춰 "operate independently over extended periods" 또는 "potentially operate for many turns" 중 하나를 정확히 인용한다.

### 3. Paperclip 어댑터 메서드명과 스케줄러 흐름

**위치**: L47

**현재**: **Heartbeat scheduler** 가 일정 주기로 깨어나 adapter `invoke()` 를 호출하면, 외부 에이전트(Claude Code/Codex 등)가 한 turn 분량의 일을 수행한 뒤 산출물을 task DB 에 기록한다

**문제**: [ACCEPT] 사실 정확성 위반. 현재 서버 어댑터 호출은 `adapter.execute(...)` 이며 `adapter.invoke()` 메서드는 아니다(`server/src/services/heartbeat.ts:7647-7663`). 또한 scheduler tick 은 직접 어댑터를 호출하기보다 agent 정책을 보고 wakeup 을 enqueue 한다(`server/src/services/heartbeat.ts:9726-9754`). `adapter.invoke` 는 실행 중 run event 이름으로 쓰인다(`server/src/services/heartbeat.ts:7635-7644`).

**제안**: "Heartbeat scheduler 가 정책에 따라 wakeup 을 enqueue 하고, 실행 경로가 run 을 claim 한 뒤 `adapter.execute(ctx)` 로 외부 런타임을 한 turn 실행한다"로 바꾼다. 로그 이벤트명을 언급하려면 `adapter.invoke` 는 eventType 이라고 분리한다.

### 4. silent-run watchdog 의 판단 신호

**위치**: L47

**현재**: **silent-run watchdog** 은 heartbeat 신호 없이 멈춘 run 을 timeout 으로 탐지해 stranded-work 회복 경로로 보낸다

**문제**: [ACCEPT] 사실 정확성 위반. Paperclip 의 silent active-run 탐지는 agent 가 보내는 별도 heartbeat 신호 부재가 아니라 output silence 를 본다. `buildRunOutputSilence()` 는 `lastOutputAt`, `processStartedAt`, `startedAt`, `createdAt` 로 silence age 를 계산하고(`server/src/services/recovery/service.ts:676-683`, `:726-760`), `scanSilentActiveRuns()` 는 running run 중 그 기준 시간이 suspicion threshold 이전인 후보를 고른다(`server/src/services/recovery/service.ts:1156-1167`). Ch03도 "출력 silence 가 진짜 평가 신호"라고 합의했다(`docs/report/03-runtime-execution.md:122-123`).

**제안**: "heartbeat 신호 없이"를 "출력 이벤트 없이" 또는 "`lastOutputAt` 기준으로 silence threshold 를 넘긴"으로 정정한다.

### 5. execution-semantics 문서 경로

**위치**: L47

**현재**: stranded-work 회복 경로(`docs/execution-semantics.md` 참조)

**문제**: [ACCEPT] 인용 완결성/사실 정확성 위반. 저장소에는 `docs/execution-semantics.md` 가 없고, 실제 문서는 `doc/execution-semantics.md` 이다(`doc/execution-semantics.md:1-9`). 현재 경로는 독자가 따라갈 수 없는 깨진 인용이다.

**제안**: `doc/execution-semantics.md` 로 고치고, 가능하면 stranded-work 관련 절 또는 Ch03의 합의 라인(`docs/report/03-runtime-execution.md:85-90`, `:100-113`)을 함께 붙인다.
