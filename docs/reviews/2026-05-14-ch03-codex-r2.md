# 챕터 03 리뷰 — 라운드 2

## 종합 판정

라운드 1 ACCEPT의 큰 방향은 반영됐다. 어댑터 계약, checkout 주체, `execute` 기반 실행, source-scoped recovery action, startup 7-pass는 본문에 들어왔다. 다만 수정 과정에서 recovery action의 유니크 키 설명과 startup 순서 해석에 새 사실 오류가 남았다.

## ACCEPT-worthy

### R2-A1. 위치: `03-runtime-execution.md:90`

- 현재: `issue_recovery_actions` 행이 `kind × status × source_issue_id × fingerprint` 로 유일성을 보장한다고 설명.
- 문제: 실제 유니크 인덱스는 두 개다. 활성 상태(`active`, `escalated`)에 대해 `(company_id, source_issue_id)` 가 유일하고, 별도 fingerprint 인덱스는 `(company_id, source_issue_id, cause, fingerprint)` 다. `kind` 는 유니크 키에 포함되지 않고, `status` 는 키 컬럼이 아니라 partial index 조건이다.
- 제안: "활성 recovery action은 source issue당 하나로 제한되고, 같은 source issue 안에서는 `cause + fingerprint` 조합으로 중복 upsert를 막는다"로 정정.
- 근거: `packages/db/src/schema/issue_recovery_actions.ts:61-66`, `server/src/services/issue-recovery-actions.ts:80-93`.

### R2-A2. 위치: `03-runtime-execution.md:100-102`

- 현재: 제목은 "Startup reconciliation — 4단계"인데 본문/코드 2는 7단계를 나열한다. 또 "순서를 바꾸면 silent run이 watchdog에 잡히기 전에 stranded 처리가 먼저 들어가는 모순"이라고 설명한다.
- 문제: 현재 구현 순서 자체가 stranded assigned work reconciliation 뒤에 silent active run scan을 수행한다. 따라서 "stranded 처리가 먼저 들어가는 것"을 모순 예시로 들면 실제 구현 순서와 충돌한다.
- 제안: 제목을 "Startup reconciliation — 7단계"로 바꾸고, 순서 설명은 "stranded assigned work와 silent active-run watchdog은 서로 다른 실패 모드를 순서대로 다룬다" 정도로 정리한다.
- 근거: `server/src/index.ts:676-717`, `server/src/index.ts:742-780`, `doc/execution-semantics.md:325-336`.

### R2-A3. 위치: `03-runtime-execution.md:64-66`

- 현재: 전이 요약에 `* → blocked`, `* → cancelled` 를 남겨 두고, "V1 전이표가 허용하는 진입 경로"라고 설명한다.
- 문제: V1 전이표는 wildcard 전이를 쓰지 않는다. `blocked` 는 `todo` 또는 `in_progress` 에서 들어가고, `cancelled` 도 `done` 에서 다시 들어가는 전이를 정의하지 않는다. 본문 설명이 "임의 상태에서 자유 전이는 아니다"라고 보강하긴 했지만, 표기 자체가 여전히 전이표와 다르게 읽힌다.
- 제안: wildcard를 제거하고 SPEC 전이표 그대로 요약한다. recovery/watchdog이 만드는 blocked 진입은 "주로 `todo`/`in_progress` 계열의 실행·대기 실패를 `blocked` 로 표면화한다"처럼 별도 문장으로 둔다.
- 근거: `doc/SPEC-implementation.md:421-426`.
