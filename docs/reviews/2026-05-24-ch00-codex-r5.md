# ch00 (00-overview.md) Round 5 리뷰 — Codex

## 1. 원칙 5 설명이 현재 회복 원시 모델을 `recovery issue`로 과도하게 좁힘

**위치**  
`docs/report/00-overview.md:58`

**현재 본문 인용**  
> `| 5 | **Surface problems, don't hide them** | 자동 회복은 보수적으로만 — 문제를 자동 봉합하지 않고 명시적인 recovery issue 로 드러낸다. |`

**문제 카테고리**  
정합성 — 엄격

**제안 수정안**  
> `| 5 | **Surface problems, don't hide them** | 자동 회복은 보수적으로만 — 문제를 자동 봉합하지 않고 명시적인 recovery action 으로 드러낸다. 별도 recovery issue 는 독립 repair work 가 필요할 때의 fallback 이다. |`

**근거**  
현재 Group A 표준과 `doc/execution-semantics.md`는 회복 원시 모델을 `recovery issue`가 아니라 `explicit recovery action`으로 둡니다. source-scoped recovery action이 기본이고, 별도 issue-backed recovery는 독립 작업이 필요할 때의 fallback입니다.

```text
doc/execution-semantics.md:172
- an open explicit recovery action that names the owner and action needed to restore liveness

doc/execution-semantics.md:174-177
### Explicit recovery actions
An explicit recovery action is a typed liveness repair path for a source issue. It is the recovery primitive; the action can be rendered directly on the source issue or backed by a separate recovery issue when the repair needs its own work item.

doc/execution-semantics.md:187-190
A source-scoped recovery action is the default form.
Use an issue-backed recovery action only when the recovery is genuinely independent work or when source-scoped handling would be unsafe or unclear.
```

`doc/SPEC-implementation.md`도 같은 구분을 둡니다.

```text
doc/SPEC-implementation.md:40
Recovery | Liveness/watchdog recovery preserves explicit ownership: retry lost execution continuity where safe, otherwise open visible source-scoped recovery actions by default, use issue-backed recovery only for independent repair work, or require human escalation

doc/SPEC-implementation.md:446
explicit recovery actions are the liveness primitive; source-scoped actions are the default form, issue-backed recovery is a fallback for independent repair work or safety boundaries
```

같은 문단의 다음 줄은 이미 `명시 recovery action :174-197`로 맞게 고쳐져 있으므로, 표 1의 원칙 설명도 같은 용어로 맞춰야 합니다.
