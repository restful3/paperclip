R03는 Claude Code의 `-p` 실행 표면과 Paperclip `claude-local` 연결 구조를 전반적으로 잘 잡고 있습니다. 다만 공식 문서의 현재 표현과 다른 직접 서술, headless 모드에서 skills가 동작하는 범위, Paperclip 파서의 retry 신호 분류를 보정해야 합니다.

### 1. 설치 명령의 curl 옵션 누락

**위치**: L15

**현재**: ``Terminal CLI — `curl https://claude.ai/install.sh | bash` 또는 Homebrew, WinGet, apt/dnf/apk로 설치하는 풀 기능 CLI.``

**문제**: [ACCEPT] 사실 정확성 문제. Claude Code overview의 macOS/Linux/WSL native install 명령은 `curl -fsSL https://claude.ai/install.sh | bash`입니다. 현재 본문은 설치 방법의 핵심은 맞지만, 검증 대상인 설치 명령을 직접 코드로 제시하면서 `-fsSL` 옵션을 누락했습니다. 출처: https://code.claude.com/docs/en/overview

**제안**: 명령을 ``curl -fsSL https://claude.ai/install.sh | bash``로 고치세요.

### 2. `headless mode` 직접 인용 근거 부재

**위치**: L24

**현재**: `공식 문서는 "CLI는 이전에 'headless mode'로 불렸으며, `-p` 플래그와 모든 CLI 옵션은 동일하게 작동한다"고 명시한다`

**문제**: [ACCEPT] 인용/사실 정확성 문제. 현재 공식 headless 문서는 “Run Claude Code programmatically” 페이지에서 `-p`/`--print`를 non-interactive 실행 플래그로 설명하고, “All CLI options work with `-p`”라고 설명합니다. 그러나 “이전에 headless mode로 불렸다”는 문장은 해당 페이지에서 확인되지 않습니다. 출처: https://code.claude.com/docs/en/headless

**제안**: 직접 인용을 제거하고 “Claude Code의 비대화 실행은 `-p`/`--print` 플래그로 진입하며, 공식 문서는 모든 CLI 옵션이 `-p`와 함께 동작한다고 설명한다”처럼 현재 문서가 확인하는 범위로 바꾸세요.

### 3. skills를 인터랙티브 전용으로 과잉 일반화

**위치**: L40

**현재**: `slash commands(`/commit` 등) 와 사용자 정의 skills 는 인터랙티브 모드 전용이며, `-p` 모드에서는 같은 의도를 평문 프롬프트로 기술해야 한다`

**문제**: [ACCEPT] 사실 정확성 문제. headless 문서가 인터랙티브 전용이라고 한 것은 “user-invoked skills like `/commit`”과 built-in commands입니다. 같은 문서는 `--bare`가 hooks, skills, plugins, MCP servers, auto memory, `CLAUDE.md` 자동탐색을 끈다고 설명하므로, 일반적인 skills 자체는 `-p`에서 항상 비활성인 것이 아니라 bare mode 여부와 설정 소스에 영향을 받습니다. 출처: https://code.claude.com/docs/en/headless

**제안**: “`/commit`처럼 사용자가 slash 형태로 호출하는 skills와 built-in commands는 인터랙티브 전용이다. 일반 skills 자동 로드는 `--bare` 사용 여부와 설정 소스에 따라 달라진다”로 정정하세요.

### 4. retryNotBefore·transient·login 신호 분류 혼동

**위치**: L75

**현재**: `parse.ts의 정규식은 5-hour/weekly usage limit, rate-limit, 503/529 등 transient upstream 에러와 로그인 요구 메시지를 식별해 paperclip 컨트롤 플레인이 재시도 시점(`retryNotBefore`)을 결정하도록 신호한다.`

**문제**: [ACCEPT] 사실 정확성 문제. `parse.ts`는 5-hour/weekly/usage reset 패턴에서만 `extractClaudeRetryNotBefore`를 계산합니다(`packages/adapters/claude-local/src/server/parse.ts:14-15`, `:355-367`). 429/503/529 등은 `isClaudeTransientUpstreamError`가 transient 여부를 판정하는 별도 신호이고(`packages/adapters/claude-local/src/server/parse.ts:12-13`, `:370-390`), 로그인 요구는 `detectClaudeLoginRequired`에서 별도로 감지되며 transient 판정에서는 제외됩니다(`packages/adapters/claude-local/src/server/parse.ts:9`, `:132-148`, `:381-386`). 현재 문장은 이 세 신호가 모두 `retryNotBefore` 결정으로 이어지는 것처럼 읽힙니다.

**제안**: “usage limit reset 문구는 `retryNotBefore` 계산에 쓰이고, 429/503/529 등은 transient upstream 에러로 분류되며, 로그인 요구는 별도 auth-required 신호로 분리된다”처럼 세 분류를 나누세요.
