# 챕터 05 리뷰 — 라운드 1

## 종합 판정

CHANGE-IMPACT의 MINOR 판정이 맞다. 큰 서버 구조 설명은 유지 가능하지만, 현재 라우트/서비스 표면과 맞지 않는 substantive 사실 오류가 있다. 특히 WebSocket 인증 계약, live event type 수, secrets/recovery-actions 확장, plugin-managed skills 서비스가 본문에 반영돼야 한다.

## ACCEPT-worthy

### R1-A1. 위치: `05-server-api.md:19`, `:89-103`

- 현재: WebSocket 라이브 채널을 "Express 서버 안에서 같은 인증/세션 스택을 재사용하는 라우트"라고 설명하고, 헤드리스 클라이언트가 `Authorization: Bearer <agent api key 또는 board api key>`로 접속한다고 쓴다.
- 문제: live events WebSocket은 Express route가 아니라 HTTP server의 `upgrade` handler로 붙는 `ws` 서버다. 인증도 `actorMiddleware`를 그대로 쓰지 않는다. 무토큰 브라우저는 `local_trusted` 또는 Better Auth session으로 처리하고, bearer token은 `agent_api_keys`만 조회한다. REST의 board API key 인증은 `actorMiddleware`에 있지만 WebSocket upgrade path에는 없다.
- 제안: "HTTP upgrade handler에서 별도 authorizeUpgrade가 local_trusted/Better Auth session/agent API key를 검증한다. 현재 WebSocket bearer token은 agent API key 경로다"로 수정한다.
- 근거: `server/src/index.ts:654-656`, `server/src/realtime/live-events-ws.ts:63-80`, `server/src/realtime/live-events-ws.ts:95-176`, `server/src/realtime/live-events-ws.ts:236-270`, `server/src/middleware/auth.ts:109-129`.

### R1-A2. 위치: `05-server-api.md:107-120`

- 현재: "표 2의 8개 type"이라고 설명한다.
- 문제: `LIVE_EVENT_TYPES`는 9개다. 표에서 `plugin.worker.crashed` / `plugin.worker.restarted`를 한 행에 합쳤지만, type 수는 9개로 세야 한다.
- 제안: "9개 type"으로 고치거나 "표는 8행이지만 worker lifecycle 행은 2개 type을 합쳐 표시한다"로 명확히 쓴다.
- 근거: `packages/shared/src/constants.ts:596-606`.

### R1-A3. 위치: `05-server-api.md:40`

- 현재: `secrets.ts` 대표 엔드포인트가 `POST /api/companies/:companyId/secrets` 하나뿐이다.
- 문제: 현재 secrets 라우트는 단순 secret CRUD를 넘어 provider listing/health/config, remote import preview/import, rotate, usage, access-events까지 확장돼 있다. CHANGE-IMPACT의 secrets route 확장 사항이 본문에 반영되지 않았다.
- 제안: secrets 행 책임을 "회사 비밀·provider vault·remote import·usage/audit" 정도로 확장하고 대표 엔드포인트에 `GET /api/companies/:companyId/secret-providers`, `GET/POST /api/companies/:companyId/secret-provider-configs`, `POST /api/companies/:companyId/secrets/remote-import`, `POST /api/secrets/:id/rotate`, `GET /api/secrets/:id/access-events`를 일부 포함한다.
- 근거: `server/src/routes/secrets.ts:22-44`, `server/src/routes/secrets.ts:79-195`, `server/src/routes/secrets.ts:228-342`, `server/src/routes/secrets.ts:380-448`.

### R1-A4. 위치: `05-server-api.md:31`, `:44`, `:53-83`

- 현재: issues route/service 설명이 checkout, status 전이, monitor, recovery를 일반적으로 언급하지만 recovery-actions API와 service를 별도로 반영하지 않는다.
- 문제: 현재 `issues.ts`에는 recovery action 조회/해결 API가 있고, service 계층에는 `issue-recovery-actions.ts`가 있다. 이는 CHANGE-IMPACT의 신규 recovery-actions 라우트와 직접 대응한다.
- 제안: `issues.ts` 대표 엔드포인트에 `GET /api/issues/:id/recovery-actions`, `POST /api/issues/:id/recovery-actions/resolve`를 추가하고, services 트리의 issue lifecycle 또는 heartbeat/recovery 군집에 `issue-recovery-actions.ts`를 명시한다.
- 근거: `server/src/routes/issues.ts:1769-1909`, `server/src/services/issue-recovery-actions.ts:1-359`.

### R1-A5. 위치: `05-server-api.md:42`

- 현재: `org-chart-svg.ts` 라우트 그룹의 대표 엔드포인트를 `GET /api/companies/:companyId/org-chart.svg`로 쓴다.
- 문제: 실제 mounted endpoint는 `agents.ts` 안의 `GET /api/companies/:companyId/org.svg`와 `GET /api/companies/:companyId/org.png`다. `org-chart-svg.ts`는 renderer helper를 export하며 독립 route로 mount되지 않는다.
- 제안: 행을 "org chart render helper / agents route"로 바꾸고 대표 엔드포인트를 `GET /api/companies/:companyId/org.svg`, `GET /api/companies/:companyId/org.png`로 수정한다.
- 근거: `server/src/routes/agents.ts:1694-1712`, `server/src/routes/agents.ts:78`, `server/src/routes/org-chart-svg.ts:703-769`, `server/src/app.ts:192`.

### R1-A6. 위치: `05-server-api.md:152-166`

- 현재: plugin SDK hook 카테고리를 lifecycle, data namespaces, events/webhooks, UI extensions, adapter plugins의 5개로 설명한다.
- 문제: 현재 server plugin host는 plugin-managed skills를 별도 service로 제공하고, `plugin-host-services.ts`에서 host services에 연결한다. Ch04에서 정리한 plugin-managed skills가 Ch05 서버 API/서비스 설명에는 빠져 있다. managed agents/projects/routines 중에서도 특히 `plugin-managed-skills.ts`는 이번 CHANGE-IMPACT 예상 누락 항목이다.
- 제안: plugin hook 카테고리에 "managed resources: agents/projects/routines/skills"를 추가하고, `plugin-managed-skills.ts`가 manifest-declared skill을 `get/reconcile/reset`으로 회사 스킬에 동기화한다는 한 줄을 넣는다.
- 근거: `server/src/services/plugin-host-services.ts:508-518`, `server/src/services/plugin-managed-skills.ts:156-359`, `packages/plugins/sdk/src/types.ts:852-862`.
