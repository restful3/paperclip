# 챕터 03 리뷰 — 라운드 1

## 종합 판정

챕터 03은 실행 모델의 큰 방향은 맞지만, 현재 구현 기준으로는 어댑터 계약, recovery action, watchdog, startup reconciliation 설명이 오래됐다. 특히 handout 합의 사항인 "장기 SPEC의 `invoke/status/cancel` 과 현재 구현의 `execute/testEnvironment` 구분"을 반영해야 하고, CHANGE-IMPACT가 지적한 source-scoped recovery action 모델이 §6에 빠져 있다.

## ACCEPT-worthy

### A1. 위치: `03-runtime-execution.md:29-43`

- 현재: "에이전트 어댑터의 3-메서드 계약"을 `invoke/status/cancel` 로 설명하고, `invoke` 만 의무라고 쓴다.
- 문제: 이는 장기 SPEC의 추상 계약이고 현재 구현 계약이 아니다. 현재 `ServerAdapterModule` 의 필수 메서드는 `execute(ctx)` 와 `testEnvironment(ctx)` 두 개이며, `status/cancel` 은 필수 인터페이스에 없다.
- 제안: 제목을 "장기 SPEC 계약과 현재 구현 계약"으로 바꾸고, 장기 SPEC은 `invoke/status/cancel`, 현재 구현은 `execute/testEnvironment` 라고 두 층으로 분리한다. 현재 런타임 설명에서는 `adapter.execute(...)` 를 기준으로 서술한다.
- 근거: `packages/adapter-utils/src/types.ts:349-352`, `server/src/services/heartbeat.ts:7647-7663`, `server/src/adapters/registry.ts:236-457`.

### A2. 위치: `03-runtime-execution.md:11`, `03-runtime-execution.md:17`

- 현재: heartbeat 흐름을 `scheduler tick → adapter invoke → agent runs...` 로 요약하고, t₁에 "어댑터가 atomic checkout으로 이슈를 잠근다"고 설명한다.
- 문제: 현재 구현에서 checkout은 어댑터가 아니라 heartbeat 서비스가 `issuesSvc.checkout(...)` 로 수행하고, 그 뒤에 서버가 `adapter.execute(...)` 를 호출한다. 이벤트명에는 `adapter.invoke` 가 남아 있지만 실제 인터페이스 메서드는 `execute` 다.
- 제안: 흐름을 "scheduler/wake claim → server-side atomic checkout → adapter.execute → agent runtime..." 로 정정한다. `adapter.invoke` 는 이벤트명 또는 장기 SPEC 용어로만 제한한다.
- 근거: `server/src/services/heartbeat.ts:6761-6816`, `server/src/services/heartbeat.ts:7628-7663`.

### A3. 위치: `03-runtime-execution.md:17`, `03-runtime-execution.md:72-75`

- 현재: t₆에 결과가 `in_review` 로 핸드오프되고 "락이 해제된다"고 설명한다. 표 1에서는 `checkout_run_id` 를 오너십 락이라고 정의한다.
- 문제: 현재 run 종료 후 `releaseIssueExecutionAndPromote()` 는 `execution_run_id` 와 실행 관련 필드를 비우지만, `checkout_run_id` 를 일반적으로 해제하지 않는다. 따라서 "락 해제"라고 쓰면 오너십 락까지 풀리는 것으로 오해된다.
- 제안: "실행 락(`execution_run_id`)은 해제되지만, 오너십/checkout 정보는 별도 규칙으로 유지·해제된다"로 정밀화한다.
- 근거: `server/src/services/heartbeat.ts:8115-8165`, `server/src/services/issues.ts:3858-3890`.

### A4. 위치: `03-runtime-execution.md:55-62`

- 현재: "어떤 상태든 → `blocked`", "어떤 상태든 → `cancelled`", "`blocked` 는 blocker가 모두 풀리면 자동으로 `in_progress` 로 돌아오는 상태"라고 설명한다.
- 문제: V1 전이표는 terminal states를 `done`/`cancelled` 로 두며, 모든 상태에서 blocked/cancelled로 가는 전이를 선언하지 않는다. 또한 blocker 해결 시 구현은 dependent agent에게 `issue_blockers_resolved` wake를 enqueue하고, 이후 checkout 경로가 `blocked` 를 `in_progress` 로 전환한다. DB 상태가 즉시 자동 전이되는 설명은 과하다.
- 제안: 전이 요약을 SPEC 전이표에 맞추고, blocked 해소는 "마지막 blocker가 `done` 이 되면 wake가 큐잉되고, 해당 run의 checkout이 성공할 때 `in_progress` 로 들어간다"로 바꾼다.
- 근거: `doc/SPEC-implementation.md:421-426`, `server/src/routes/issues.ts:3543-3566`, `server/src/services/heartbeat.ts:6803-6816`, `server/src/services/issues.ts:408-437`.

### A5. 위치: `03-runtime-execution.md:62`, `03-runtime-execution.md:84`

- 현재: 회복 경로를 "explicit recovery issue" 로 부르고, stranded 재시도 실패 후 이슈를 `blocked` 로 옮기고 보이는 코멘트를 남긴다고 설명한다.
- 문제: 현재 계약은 source-scoped `issue_recovery_actions` 가 기본 recovery primitive다. comment는 evidence일 수 있지만 recovery path 자체가 아니다. issue-backed recovery는 독립 작업이 필요할 때의 fallback이다.
- 제안: §6에 source-scoped recovery action 단락을 추가하고, `kind` / `status` / `fingerprint`, `source_issue_id`, `owner_*`, `wake_policy`, `monitor_policy` 를 중심으로 설명한다. "explicit recovery issue" 는 "explicit recovery action" 으로 바꾸고 issue-backed 형태는 예외로 둔다.
- 근거: `doc/execution-semantics.md:172-197`, `packages/db/src/schema/issue_recovery_actions.ts:16-67`, `server/src/services/issue-recovery-actions.ts:19-38`, `server/src/services/issue-recovery-actions.ts:160-257`.

### A6. 위치: `03-runtime-execution.md:77-85`

- 현재: stranded recovery 규칙을 "자동으로 한 번만 wake를 재큐잉하고, 또 stranded이면 blocked + visible comment" 로만 설명한다.
- 문제: 현재 구현은 retry exhausted 이후 source-scoped recovery action을 upsert하고, recovery owner를 정해 source issue를 `blocked` 로 만들며, 필요하면 `source_scoped_recovery_action` wake를 큐잉한다. visible comment는 action id와 owner를 보여 주는 표면일 뿐이다.
- 제안: "한 번 retry" 뒤의 두 번째 단계는 "blocked + source-scoped recovery action + owner/wake policy" 로 확장한다.
- 근거: `doc/execution-semantics.md:302-321`, `server/src/services/recovery/service.ts:1618-1666`, `server/src/services/recovery/service.ts:1669-1698`, `server/src/services/recovery/service.ts:1828-1880`.

### A7. 위치: `03-runtime-execution.md:86-92`

- 현재: silent watchdog이 `suspicious` 에서 medium-priority review issue를 만들고, `critical` 에서 high-priority로 격상한다고만 설명한다. 인용문도 "review issue" 중심이다.
- 문제: 현재 실행 의미론은 "watchdog recovery action" 을 중심으로 설명하고, issue-backed 구현은 `stale_active_run_evaluation` 이슈를 사용하는 형태다. 현재 구현은 여전히 evaluation issue를 만들지만, CHANGE-IMPACT 기준으로는 recovery action/source-scoped 모델을 반영해야 한다.
- 제안: "watchdog 신호는 recovery action으로 표면화되며, issue-backed 평가가 필요한 경우 `stale_active_run_evaluation` 이슈를 만든다"로 조정한다. 인용문도 현재 `doc/execution-semantics.md` §10의 recovery action 표현으로 갱신한다.
- 근거: `doc/execution-semantics.md:338-361`, `packages/shared/src/constants.ts:204-223`, `server/src/services/recovery/service.ts:672-673`, `server/src/services/recovery/service.ts:1026-1125`, `server/src/services/recovery/service.ts:1236-1266`.

### A8. 위치: `03-runtime-execution.md:94-107`

- 현재: startup reconciliation을 4-pass로 고정하고 `reap → resume queued → stranded → silent watchdog` 만 나열한다.
- 문제: 현재 startup/periodic 경로에는 orphan reap 뒤 `promoteDueScheduledRetries()` 가 들어가고, stranded reconciliation 뒤 `reconcileIssueGraphLiveness()`, `scanSilentActiveRuns()`, `reconcileProductivityReviews()` 가 이어진다. 본문 4-pass는 현재 코드의 startup recovery 전체 순서를 설명하기에는 누락이 있다.
- 제안: 코드 2를 "핵심 4-pass" 로 좁히거나, 현재 구현 순서대로 `reap → promote scheduled retries → resume queued → stranded assigned issues → issue-graph liveness → silent active runs → productivity reviews` 로 갱신한다.
- 근거: `server/src/index.ts:676-717`, `server/src/index.ts:742-780`, `doc/execution-semantics.md:325-336`.

- 문제: 현재 구현의 `ServerAdapterModule` 에는 필수 `status()` 메서드가 없다. 이 문장은 A1과 같은 stale SPEC 용어다.

## DEFER

- `03-runtime-execution.md:21-25` 의 외부 그림 번호 `그림 5-A` 는 handout의 그림 번호 통합 원칙과 어긋난다. 다만 이번 라운드는 runtime 사실 오류가 더 크므로, 번호 정리는 본문 사실 수정과 함께 일괄 처리해도 된다.
- `03-runtime-execution.md:17` 의 "시간 축이 단조 증가하므로 한 라이프라인 안에서 동시 실행이 일어나지 않는다"는 다이어그램 해석이다. 실제 동시성 보장은 DB checkout과 run claim 쪽에서 봐야 하므로, 그림 문장 정리는 후속 표현 효율성 라운드로 미룬다.
