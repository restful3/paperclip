## 5. middleware 설명이 실제 권한 검사 위치와 redaction/error 처리 위치를 잘못 묶음

**위치**  
`docs/report/05-server-api.md` §6 `middleware/ — 회사 경계와 redaction`

**현재 본문 인용**  
> 라우트 본체에 도달하기 전 미들웨어가 다음을 강제한다.  
> 1. 회사 멤버십 — URL 의 `:companyId` 가 현재 주체에 보이는지 검사  
> 2. 에이전트 키 격리 — `agent_api_keys` 는 자기 회사만, 자기 에이전트 본인의 권한 범위만  
> 3. 로그 redaction — 비밀 값(secrets, env)이 활동 로그에 새지 않도록 마스킹  
> 4. 에러 매핑 — 4xx / 409(conflict) / 422(validation) / 500 을 일관되게 변환

**문제 카테고리**  
정합성 — 엄격

**제안 수정안**  
> `middleware/`는 주로 actor 해석, board mutation guard, private hostname guard, validation, HTTP logging, error mapping을 담당한다. 회사 접근 제어는 전역 pre-route middleware가 아니라 대부분 라우트 내부의 `assertCompanyAccess(...)` / `assertBoardOrgAccess(...)` helper에서 수행된다. 활동 로그 redaction도 middleware가 아니라 `logActivity(...)` 서비스 내부에서 `sanitizeRecord`와 `redactCurrentUserValue`를 거쳐 수행된다. `errorHandler`는 라우트 뒤에 붙는 Express error middleware다.

**근거**  
앱 조립부에서 라우트 전에 붙는 것은 `privateHostnameGuard`, `actorMiddleware`, API 라우터 내부의 `boardMutationGuard`입니다.
