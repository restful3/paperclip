# 챕터 01 리뷰 — 라운드 1

## 종합 판정
챕터 구조는 이해하기 쉽지만, 현재 `master` 기준으로 오래된 사실이 꽤 남아 있어 ACCEPT는 어렵습니다. 가장 큰 문제는 챕터 00에서 합의한 어댑터 12개/`execute`+`testEnvironment` 구현 계약과 정면으로 충돌하는 부분이며, 패키지 의존 표도 실제 `package.json`과 맞지 않는 항목이 있습니다.

## ACCEPT-worthy 지적 (실제 수정이 필요한 항목)
- **[1] built-in adapter 수와 목록이 stale**
  - 위치: `01-architecture.md:39`
  - 현재: "현재 upstream `master`의 built-in adapter type은 11개다. 이 중 `claude-local`, `codex-local`, `cursor-local`, `gemini-local`, `opencode-local`, `pi-local`, `acpx-local`, `openclaw-gateway`는 ..."
  - 문제: 현재 서버 레지스트리 기준 built-in adapter는 12개이며 `cursor_cloud`/`cursor-cloud`가 빠져 있습니다.
  - 제안: "현재 서버 레지스트리 기준 built-in adapter type은 12개다. 패키지형 어댑터는 `acpx-local`, `claude-local`, `codex-local`, `cursor-cloud`, `cursor-local`, `gemini-local`, `opencode-local`, `pi-local`, `openclaw-gateway`이고, `hermes_local`은 `hermes-paperclip-adapter`, `process`/`http`는 서버 내장 generic adapter다."로 수정하세요.
  - 근거: `server/src/adapters/registry.ts:11-18`, `server/src/adapters/registry.ts:62-68`, `server/src/adapters/registry.ts:116-132`, `server/src/adapters/registry.ts:480-493`

- **[1] 어댑터 표준 트리의 서버 계약이 현재 구현과 불일치**
  - 위치: `01-architecture.md:43-49`
  - 현재: "`src/server/index.ts   # invoke / status / cancel + spawn 로직`"
  - 문제: 장기 SPEC의 `invoke/status/cancel`이 아니라 현재 구현의 `ServerAdapterModule` 필수 계약은 `execute(ctx)`와 `testEnvironment(ctx)`입니다.
  - 제안: 주석을 "`src/server/index.ts   # execute / testEnvironment + 런타임 실행·검증 로직`"으로 바꾸고, 본문에 "장기 SPEC은 `invoke/status/cancel`이지만 구현은 `execute`/`testEnvironment`"라고 짧게 덧붙이세요.
  - 근거: `packages/adapter-utils/src/types.ts:349-352`, `doc/SPEC.md:207-215`

- **[1] 표 1의 의존 관계가 실제 package.json과 다름**
  - 위치: `01-architecture.md:25-33`
  - 현재: "`server/` ... 의존 db · shared · adapters · mcp-server · plugins", "`packages/adapters/*` ... adapter-utils · shared", "`packages/adapter-utils` ... shared", "`packages/plugins` ... shared · DB"
  - 문제: `server`는 `@paperclipai/mcp-server`에 의존하지 않습니다. adapter 패키지들은 대체로 `@paperclipai/adapter-utils`에 의존하고 `@paperclipai/shared`를 직접 의존하지 않습니다. `adapter-utils`도 `@paperclipai/shared` 의존성이 없습니다. `packages/plugins`는 단일 패키지가 아니라 `sdk`, examples, sandbox providers 등의 워크스페이스 묶음이며 런타임 상당 부분은 `server/src/services/plugin-*`에 있습니다.
  - 제안: 표를 실제 패키지 단위로 재작성하세요. 예: `packages/plugins/sdk`는 plugin SDK, `packages/plugins/*`는 예제/프로바이더/도구 패키지, plugin host runtime은 `server/src/services/plugin-*`로 분리.
  - 근거: `server/package.json:45-79`, `packages/adapters/claude-local/package.json:55-57`, `packages/adapter-utils/package.json:37-45`, `packages/plugins/sdk/package.json:103-113`, `server/src/app.ts:46-59`

- **[1] `packages/mcp-server` 설명의 DB 의존 표현이 부정확**
  - 위치: `01-architecture.md:30`
  - 현재: "`packages/mcp-server` ... 의존 shared · DB(REST 경유)"
  - 문제: MCP server 패키지의 실제 dependency는 `@modelcontextprotocol/sdk`, `@paperclipai/shared`, `zod`뿐이며 DB 패키지를 직접 의존하지 않습니다. 동작도 `PAPERCLIP_API_URL`/`PAPERCLIP_API_KEY`로 REST API를 호출합니다.
  - 제안: 의존 열을 "`shared · REST API(fetch)`" 정도로 바꾸고 DB 의존은 제거하세요.
  - 근거: `packages/mcp-server/package.json:45-49`, `packages/mcp-server/src/config.ts:23-34`, `packages/mcp-server/src/client.ts:83-95`

- **[1] dev 서버 포트를 고정값처럼 설명함**
  - 위치: `01-architecture.md:58`, `01-architecture.md:63`
  - 현재: "`pnpm dev` ... 같은 포트(`3100`)에서 서빙", "`pnpm dev # API + UI 모두 http://localhost:3100`"
  - 문제: 기본 요청 포트는 3100이지만 서버는 `detect-port`로 사용 가능한 포트를 선택합니다. 3100이 이미 사용 중이면 다음 free port로 올라갑니다.
  - 제안: "`기본 요청 포트는 3100이며, 사용 중이면 서버가 다음 사용 가능한 포트를 선택한다. UI는 선택된 API 서버 포트에서 same-origin으로 제공된다.`"로 수정하세요.
  - 근거: `server/src/config.ts:294`, `server/src/index.ts:473-475`, `server/src/index.ts:629-630`, `doc/DEVELOPING.md:35-44`

- **[1] 인스턴스 디렉터리 레이아웃이 현재 문서보다 축약되어 표준처럼 보임**
  - 위치: `01-architecture.md:101-113`
  - 현재: "`config.json`, `db/`, `data/storage/`, `secrets/master.key`"
  - 문제: 현재 개발 문서의 local instance layout에는 `.env`, `data/backups/`, `logs/`, `workspaces/<agent-id>/`, `projects/`, `companies/<company-id>/codex-home/`도 포함됩니다. 지금 표는 "표준 디렉터리 레이아웃"이라고 하기에는 핵심 런타임 경로를 많이 생략합니다.
  - 제안: 코드 3을 `doc/DEVELOPING.md`의 현재 레이아웃과 맞추거나, "핵심 경로만 추리면"이라고 명시하세요.
  - 근거: `doc/DEVELOPING.md:160-177`, `doc/DEVELOPING.md:196-200`, `doc/DEVELOPING.md:481-483`

- **[1] fork 포트 동작을 일반 인스턴스 모델처럼 씀**
  - 위치: `01-architecture.md:115`
  - 현재: "한 머신에서 여러 인스턴스를 병렬로 운영할 수 있다(`HenkDz/paperclip` 같은 포크는 포트 3101+ 로 자동 자리잡는다)."
  - 문제: 현재 upstream 코드의 일반 동작은 특정 fork명을 기준으로 하지 않고 `detect-port`로 요청 포트가 바쁠 때 사용 가능한 포트를 고르는 방식입니다. fork-specific 예시는 본문 사실 주장으로는 불필요하고 근거도 코드가 아니라 로컬 AGENTS 지시입니다.
  - 제안: "한 머신에서 여러 인스턴스를 병렬로 운영할 때는 `PAPERCLIP_HOME`/`PAPERCLIP_INSTANCE_ID` 또는 worktree instance를 분리하고, 포트는 사용 가능 포트로 조정된다."로 바꾸세요.
  - 근거: `doc/DEVELOPING.md:179-190`, `doc/DEVELOPING.md:226-240`, `server/src/index.ts:473-475`, `server/src/index.ts:629-630`

- **[5] 사실 주장 대부분에 line citation이 없음**
