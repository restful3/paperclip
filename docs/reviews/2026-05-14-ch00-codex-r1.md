# 챕터 00 리뷰 — 라운드 1

## 종합 판정
전반적인 구조와 학습용 흐름은 좋지만, 현재 라운드에서는 ACCEPT가 어렵습니다. 가장 큰 우려는 현재 `master` 코드와 맞지 않는 수치/인터페이스 단정입니다. 특히 built-in adapter 수, 어댑터 계약, README 항목 수가 실제 파일과 다르며, 많은 사실 주장에 `path:line` 근거가 없어 독자가 검증 경로를 따라가기 어렵습니다.

## ACCEPT-worthy 지적 (실제 수정이 필요한 항목)
- **[1] README 문제 목록이 5개가 아니라 6개**
  - 위치: `00-overview.md:17-23`
  - 현재: "저장소 README와 `doc/GOAL.md`가 명시하는 출발점은 다음 다섯 가지다."
  - 문제: 현재 README의 "What goes wrong when you use raw agents" 표는 6개 문제를 제시하며, 리포트는 마지막 "아이디어→repo→Claude Code→babysit" 흐름을 누락했습니다.
  - 제안: "README는 여섯 가지 운영 문제를 든다."로 바꾸고 6번째 항목을 추가하거나, "이 장에서는 그중 다섯 가지를 추려 보면"처럼 축약임을 명시하세요.

- **[1] built-in adapter 수와 목록이 현재 코드와 불일치**
  - 위치: `00-overview.md:29`
  - 현재: "현재 upstream `master` 기준 11개의 built-in adapter type ... 9개는 구체 런타임(... Hermes)을 향하고, 2개는 범용 `process`/`http` 백업 통로다."
  - 문제: 서버 레지스트리 기준 현재 built-in set은 12개이며 `acpx_local`, `cursor_cloud`, `hermes_local`, `process`, `http`를 모두 포함합니다. 반면 shared 상수의 11개 목록은 `hermes_local`이 없고 `acpx_local`, `cursor_cloud`가 있습니다. 따라서 "11개/9개+2개/Hermes 포함"은 현재 코드의 어느 단일 목록과도 맞지 않습니다.
  - 제안: "서버 레지스트리 기준 built-in adapter는 12개(`acpx_local`, `claude_local`, `codex_local`, `cursor_cloud`, `cursor`, `gemini_local`, `openclaw_gateway`, `opencode_local`, `pi_local`, `hermes_local`, `process`, `http`)다."처럼 기준을 명확히 쓰세요. shared 타입 목록까지 설명하려면 별도 문장으로 `AGENT_ADAPTER_TYPES`와의 차이를 언급해야 합니다.
  - 근거: `server/src/adapters/builtin-adapter-types.ts:4-17`, `server/src/adapters/registry.ts:480-493`, `packages/shared/src/constants.ts:30-42`

- **[1] 현재 코드의 어댑터 계약은 `invoke/status/cancel`이 아님**

  - 현재: "어댑터의 최소 계약은 `invoke / status / cancel` 3개 메서드뿐이며"
  - 문제: 이는 `doc/SPEC.md`의 장기 설계 문구에는 있지만, 현재 코드의 `ServerAdapterModule` 필수 메서드는 `execute`와 `testEnvironment`입니다. `detectModel`, `listSkills`, `syncSkills` 등은 선택 필드입니다.
  - 제안: "장기 SPEC은 `invoke/status/cancel`을 최소 계약으로 설명하지만, 현재 구현의 서버 어댑터 모듈은 필수 `execute`와 `testEnvironment`를 중심으로 동작한다."로 수정하세요.
  - 근거: `doc/SPEC.md:207-215`, `packages/adapter-utils/src/types.ts:349-389`

- **[1] "모든 issue가 goal에 도달해야 한다"는 현재 구현 불변식이 아님**
  - 위치: `00-overview.md:56`
  - 현재: "모든 issue는 부모 사슬을 따라 회사 골(혹은 initiative)에 도달해야 한다."
  - 문제: 제품 원칙으로는 맞지만, 현재 DB/API 구현은 `issues.goal_id`와 `issues.parent_id`를 nullable로 두고 create/update validator도 둘 다 optional/nullable입니다. 기본 goal fallback도 goal이 없으면 `null`을 반환할 수 있습니다.
  - 제안: "제품 원칙상 issue는 goal 계층에 연결되도록 설계되지만, 현재 구현은 `goal_id`/`parent_id`를 nullable로 두고 생성 시 fallback을 시도한다."처럼 원칙과 구현을 분리하세요.
  - 근거: `doc/PRODUCT.md:57`, `packages/db/src/schema/issues.ts:25-29`, `packages/shared/src/validators/issue.ts:304-308`, `server/src/services/issue-goal-fallback.ts:3-12`
