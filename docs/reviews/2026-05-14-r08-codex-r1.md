외부 문서 인용(Vite, shadcn/ui, TanStack Query)과 Express 의 Vite middleware mode 설명은 원문 및 코드와 대체로 일치합니다. 다만 §5의 Paperclip 적용 문장에는 실제 UI 코드의 책임 배치와 인증 표현이 섞인 substantive 오류가 남아 있습니다.

### 1. TanStack Query 훅 위치

**위치**: L43

**현재**: `ui/src/api/` 에 TanStack Query 훅과 서버 상태 페치 레이어

**문제**: [ACCEPT] 사실 정확성 위반. `ui/src/api/client.ts:15-50` 은 `/api` fetch wrapper 와 HTTP 메서드 래퍼이고, TanStack Query 훅을 제공하지 않는다. QueryClient 는 `ui/src/main.tsx:31-38` 에서 생성되고, 실제 `useQuery`/`useMutation` 사용은 예컨대 `ui/src/context/CompanyContext.tsx:10,72-85,128-139` 처럼 context/pages/components 쪽에 있다. 따라서 `api/` 를 "TanStack Query 훅" 위치로 쓰면 구조 설명이 틀린다.

**제안**: "`ui/src/api/` 에 서버 API fetch 레이어, `ui/src/context/`·`ui/src/pages/`·`ui/src/components/` 에 TanStack Query 훅과 화면 상태가 분산돼 있다"처럼 책임을 분리해 쓰는 편이 맞다.

### 2. CSRF 동작 주장

**위치**: L43

**현재**: 그래서 외부 환경 변수 없이 쿠키·CSRF 가 자연스럽게 동작한다.

**문제**: [ACCEPT] 사실 정확성 위반. 같은 origin 이라 쿠키 기반 요청이 단순해지는 것은 맞고, `ui/src/api/client.ts:22-25` 는 `credentials: "include"` 로 쿠키를 포함한다. 하지만 현재 확인 가능한 구현은 Better Auth 쿠키 설정(`server/src/auth/better-auth.ts:40-44`)과 trusted origin 설정(`server/src/auth/better-auth.ts:64-90`)이며, `server/src`/`ui/src`/`packages/shared` 에서 `csrf|CSRF|xsrf|XSRF` 심볼은 발견되지 않는다. 단일 origin 은 CORS와 쿠키 디버깅을 줄여 주지만, 별도 CSRF 보호가 "자연스럽게 동작"한다고 쓰기에는 코드 근거가 없다.

**제안**: "그래서 CORS 설정 없이 쿠키 기반 세션 요청이 단순해진다"로 축소하거나, 실제 CSRF 방어가 별도로 존재한다면 해당 구현 path:line 을 인용해 보강한다.
