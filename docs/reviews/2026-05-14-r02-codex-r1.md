R02는 MCP의 큰 구조와 Paperclip MCP server의 위치를 대체로 정확히 설명합니다. 다만 생태계 수치, ChatGPT 도입 날짜의 인용 근거, transport 보안 규범의 강도, Paperclip MCP와 어댑터의 책임 경계에서 substantive 수정이 필요합니다.

### 1. ChatGPT MCP 도입 날짜 인용 근거 불일치

**위치**: L13

**현재**: `2025년 9월 10일 ChatGPT 에 개발자 모드(beta) 로 읽기/쓰기 MCP 클라이언트 지원을 정식 도입했다 ([InfoQ, 2025-10](https://www.infoq.com/news/2025/10/chat-gpt-mcp/)).`

**문제**: [ACCEPT] 인용 완결성/사실 검증 문제. InfoQ 기사는 2025-10-13 기사이며 “full MCP support”, “read and write actions”, “beta under Developer Mode”는 뒷받침하지만, 본문이 쓰는 정확한 2025-09-10 도입일은 해당 기사 본문에서 확인되지 않습니다. 같은 문장 안의 “OpenAI가 2025년 3월 자사 제품 전반에 MCP를 채택”도 현재 붙은 InfoQ 인용만으로는 직접 뒷받침되지 않습니다. 출처: https://www.infoq.com/news/2025/10/chat-gpt-mcp/

**제안**: exact date를 유지하려면 2025-09-10 OpenAI 발표/공식 문서 출처를 별도로 붙이고, InfoQ는 “읽기/쓰기 MCP client 지원이 Developer Mode beta로 제공됨”의 근거로만 쓰세요. 3월 OpenAI 채택 주장도 별도 출처를 붙이거나 “OpenAI는 2025년 3월 MCP 지원 계획을 발표했고…”처럼 범위를 낮추세요.

### 2. Streamable HTTP 보안 규범 강도 오기

**위치**: L43

**현재**: `DNS rebinding 방지를 위해 server 는 Origin 헤더를 검증하고 로컬 실행 시 127.0.0.1 에만 바인딩하라고 명세는 요구한다`

**문제**: [ACCEPT] 사실 정확성 문제. MCP transport 명세에서 `Origin` 검증은 `MUST`지만, 로컬 실행 시 localhost/127.0.0.1 바인딩은 `SHOULD`입니다. 둘을 모두 “요구한다”로 묶으면 표준의 규범 강도를 잘못 전달합니다. 출처: https://modelcontextprotocol.io/docs/concepts/transports

**제안**: “server는 `Origin` 헤더를 검증해야 하며(MUST), 로컬 실행 시에는 가능한 한 localhost/127.0.0.1에만 바인딩하도록 권고된다(SHOULD)”처럼 MUST/SHOULD를 분리하세요.

### 3. Zylos 생태계 수치 불일치

**위치**: L51

**현재**: `공개 레지스트리 등록 server 수는 2025년 1분기 약 1,200 개에서 2026년 4월 9,400+ 까지 성장했다`

**문제**: [ACCEPT] 사실 정확성 문제. 인용한 Zylos 글은 2026-03-08 기준 “community registries now index over 18,000 MCP servers”라고 하고, 표에서도 Glama.ai 18,374 / MCP.so 18,250+를 제시합니다. 현재 본문의 “1,200 → 9,400+” 및 “2026년 4월” 수치는 이 URL에서 확인되지 않습니다. 출처: https://zylos.ai/research/2026-03-08-mcp-remote-evolution-streamable-http-enterprise-adoption

**제안**: 해당 URL을 유지한다면 “2026년 3월 기준 커뮤니티 레지스트리는 18,000+ MCP server를 색인한다”로 고치세요. 1,200→9,400+ 수치를 쓰려면 그 수치를 실제로 제시하는 별도 출처를 붙이세요.

### 4. Paperclip MCP와 어댑터 책임 경계 혼동

**위치**: L57

**현재**: `어댑터 측은 표준 MCP 등록만 수행하면 동일한 task·comment·document·approval 조작 능력을 얻는다.`

**문제**: [ACCEPT] 사실 정확성/리포트 일관성 문제. 현재 코드에서 `packages/mcp-server`는 `McpServer`와 `StdioServerTransport`로 standalone stdio MCP server를 만들고(`packages/mcp-server/src/index.ts:7-29`), `PaperclipApiClient`가 REST API로 위임합니다(`packages/mcp-server/src/client.ts:78-113`). 반면 어댑터 실행 계약은 `execute` / `testEnvironment` 중심이며, Ch04는 SKILL/MCP를 어댑터와 “수직 직교”한 도메인 통로로 정리합니다(`docs/report/04-adapters-and-skills.md:145-155`). 따라서 “어댑터 측이 MCP 등록만 수행”한다는 표현은 MCP server와 adapter의 책임 경계를 흐립니다.

**제안**: “MCP 호환 호스트/에이전트는 Paperclip MCP server를 표준 방식으로 등록하면, 어댑터 종류와 무관하게 task·comment·document·approval 도구를 호출할 수 있다”처럼 주체를 MCP client/host 쪽으로 바꾸세요.
