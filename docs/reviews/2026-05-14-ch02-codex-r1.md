# 챕터 02 리뷰 — 라운드 1

## 종합 판정

현재 챕터 02는 설명 구조는 좋지만, 데이터 모델 장에서 가장 중요한 ER 관계·nullable 여부·마이그레이션 경로가 현재 `master` 와 여러 군데 어긋난다. 특히 `issues` 와 `heartbeat_runs` 사이에 직접 FK가 없다는 설명, `cost_events` 의 다섯 FK가 모두 nullable이라는 설명, 마이그레이션 생성 위치가 `packages/db/drizzle/` 라는 설명은 즉시 고쳐야 한다.

## ACCEPT-worthy

### A1. 위치: `02-data-model.md:11`

- 현재: `packages/db/src/schema/` 에는 **74개의 Drizzle 테이블 파일** 이 있고 총 80여 개 테이블 정의가 있다고 설명.
- 문제: 현재 스키마는 `pgTable(` 을 포함하는 파일이 78개이고, `pgTable` 정의는 86개다. `index.ts` 도 78줄까지 현재 스키마 export를 나열하며, 한 파일에서 여러 테이블을 export하는 경우가 있다.
- 제안: "현재 `pgTable` 정의는 86개이고, 이를 78개 스키마 파일이 나누어 가진다"처럼 정확히 쓰거나, 숫자 변동을 피하려면 "80개를 넘는 테이블 정의"로 좁힌다.
- 근거: `packages/db/src/schema/index.ts:1-78`, `packages/db/src/schema/auth.ts:3-40`, `packages/db/src/schema/routines.ts:22-128`, `packages/db/src/schema/plugin_database.ts:23-50`, `packages/db/src/schema/plugin_jobs.ts:30-71`.

### A2. 위치: `02-data-model.md:17`, `02-data-model.md:33`

- 현재: `issues` 와 `heartbeat_runs` 사이에는 직접 FK가 없고, 둘은 `activity_log.run_id` 를 다리로 연결된다고 설명.
- 문제: 현재 `issues` 는 `checkout_run_id` 와 `execution_run_id` 두 컬럼으로 `heartbeat_runs.id` 를 직접 참조한다. `activity_log.run_id` 도 FK지만 유일한 연결 경로가 아니다.
- 제안: "`issues` 는 `checkout_run_id` / `execution_run_id` 로 실행 회차를 직접 가리키고, `activity_log.run_id` 는 실행 중 발생한 감사 이벤트를 같은 run 축으로 묶는다"로 정정.
- 근거: `packages/db/src/schema/issues.ts:37-38`, `packages/db/src/schema/heartbeat_runs.ts:6-18`, `packages/db/src/schema/activity_log.ts:16-18`.

### A3. 위치: `02-data-model.md:17`, `02-data-model.md:32`

- 현재: `cost_events` 의 `agent_id` · `issue_id` · `project_id` · `goal_id` · `heartbeat_run_id` 다섯 외래키가 모두 nullable이라고 설명.
- 문제: `agent_id` 는 `.notNull()` 이며 nullable이 아니다. nullable인 것은 `issue_id`, `project_id`, `goal_id`, `heartbeat_run_id` 이다.
- 제안: "`agent_id` 는 필수이고, issue/project/goal/heartbeat run 축은 선택적으로 붙는다"로 수정.
- 근거: `packages/db/src/schema/cost_events.ts:13-18`.

### A4. 위치: `02-data-model.md:15` / `docs/report/assets/fig-04-er-core.svg`

- 현재: 그림 4의 ER 라벨이 `goals.parent_goal_id`, `projects.budget_monthly_cents`, `budget_policies.action_on_breach` 를 현재 컬럼처럼 표시.
- 문제: 현재 `goals` 의 self FK 컬럼은 `parent_id` 이고, `projects` 에는 `budget_monthly_cents` 컬럼이 없다. `budget_policies` 도 `action_on_breach` 대신 `hard_stop_enabled`, `notify_enabled`, `warn_percent` 등으로 정책을 표현한다.
- 제안: 그림 라벨을 `parent_id (FK self)`, 프로젝트의 실제 컬럼(`execution_workspace_policy`, `env`, `archived_at` 등) 또는 예산 관련 라벨 제거, `hard_stop_enabled / notify_enabled` 중심으로 정정.
- 근거: `docs/report/assets/fig-04-er-core.svg:39-44`, `docs/report/assets/fig-04-er-core.svg:48-55`, `docs/report/assets/fig-04-er-core.svg:128-132`, `packages/db/src/schema/goals.ts:21-22`, `packages/db/src/schema/projects.ts:11-23`, `packages/db/src/schema/budget_policies.ts:9-17`.

### A5. 위치: `02-data-model.md:52`, `02-data-model.md:60`

- 현재: atomic checkout 실제 코드 위치를 `server/src/services/issues.ts:3409~3525` 로 적고, 예시 status에 `open` 을 포함.
- 문제: 현재 checkout 구현은 `server/src/services/issues.ts:3593-3736` 에 있다. 또한 현재 이슈 status 집합에는 `open` 이 없고, `backlog`, `todo`, `in_progress`, `in_review`, `done`, `blocked`, `cancelled` 만 있다.
- 제안: 코드 위치를 현재 라인대로 갱신하고, 예시 status는 `backlog/todo/blocked/in_review` 또는 SPEC 예시의 `todo/backlog/blocked/in_review` 로 바꾼다.
- 근거: `server/src/services/issues.ts:3593-3736`, `packages/shared/src/constants.ts:127-135`, `doc/SPEC-implementation.md:546-554`.

### A6. 위치: `02-data-model.md:93`, `02-data-model.md:108`

- 현재: 스키마 변경 후 SQL 마이그레이션은 명시 적용만 가능하고 "자동 적용은 없다"고 설명하며, 생성된 마이그레이션이 `packages/db/drizzle/` 아래에 누적된다고 설명.
- 문제: `drizzle.config.ts` 의 출력 경로는 `./src/migrations` 이고, 실제 마이그레이션은 `packages/db/src/migrations/` 아래에 있다. 또한 dev/서버 부트 경로에는 자동 적용 조건이 있다. `pnpm dev:once` 는 pending local migration을 기본 자동 적용하고, 첫 embedded PostgreSQL setup 때도 pending migration 자동 적용 로그가 있다.
- 제안: "생성 위치는 `packages/db/src/migrations/` 이며, 수동 적용 명령은 `pnpm db:migrate` 다. 다만 dev/server startup 경로에는 pending/empty-db migration 자동 적용 경로가 있으므로 운영 문맥에 따라 동작이 다르다"로 정정.
- 근거: `packages/db/drizzle.config.ts:1-6`, `packages/db/package.json:38-43`, `doc/DEVELOPING.md:42-66`, `doc/DATABASE.md:17-24`, `server/src/index.ts:167-182`, `server/src/index.ts:433`.
