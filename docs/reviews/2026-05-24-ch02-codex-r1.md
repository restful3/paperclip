**문제 카테고리**  
정합성

**제안 수정안**  
> `packages/db/src/schema/` 에는 **88개의 `pgTable()` 정의가 79개 `pgTable` 보유 스키마 파일** 에 분산되어 있다(`packages/db/src/schema/index.ts:1-79`).

또는 파일 수를 전체 `.ts` 기준으로 말하려면:

> `packages/db/src/schema/` 에는 **88개의 `pgTable()` 정의가 80개 스키마 `.ts` 파일(그중 `pgTable` 보유 파일 79개)** 에 분산되어 있다(`packages/db/src/schema/index.ts:1-79`).

**근거**  
Group A 합의 표준은 `pgTable` 88개다. 실제 `rg -n "pgTable\\(" packages/db/src/schema | wc -l` 결과도 88이다. `packages/db/src/schema/index.ts` 는 현재 79행까지 export가 있으며, `cloudUpstreamConnections`/`cloudUpstreamRuns` 는 `packages/db/src/schema/index.ts:5` 에서 export된다. `routines.ts` 는 한 파일에 4개 테이블(`packages/db/src/schema/routines.ts:22`, `:57`, `:90`, `:125`)을 정의하고, `cloud_upstreams.ts` 도 2개 테이블(`packages/db/src/schema/cloud_upstreams.ts:4`, `:47`)을 정의하므로 "파일 수 = 테이블 수" 식의 카운트는 맞지 않는다.

## 2.

**위치**  
`docs/report/02-data-model.md:17`, `docs/report/02-data-model.md:21`

**현재 본문 인용**  
> 이 그림은 **모든 비즈니스 데이터가 `company_id`로 스코프** 되며...

> 80여 개 테이블 중 60여 개가 `company_id uuid not null references companies.id` 컬럼을 들고 있다.

**문제 카테고리**  
정합성

**제안 수정안**  
> 이 그림은 **핵심 회사 도메인 데이터가 `company_id`로 스코프** 되며...

> 88개 `pgTable()` 정의 중 67개가 `company_id` 컬럼을 들고 있다. 다만 일부 플러그인/인스턴스/파생 테이블은 `plugin_id`, `user_id`, 상위 company-scoped FK 등 다른 축으로 귀속된다.

**근거**  
Group A 합의 표준은 `company_id` 보유 테이블 67개다. 실제 테이블 단위 카운트도 67개다. 반대로 모든 비즈니스/확장 테이블이 직접 `company_id`를 갖지는 않는다. 예를 들어 `plugin_database_namespaces` 는 `plugin_id` 중심 테이블이고 `company_id`가 없다(`packages/db/src/schema/plugin_database.ts:23-36`), `plugin_migrations` 도 `plugin_id`/`namespace_name` 중심이다(`packages/db/src/schema/plugin_database.ts:50-66`). `company_secret_versions` 역시 직접 `company_id` 없이 `secret_id` 로 `company_secrets` 에 귀속된다(`packages/db/src/schema/company_secret_versions.ts:5-20`; 상위 `company_secrets.company_id` 는 `packages/db/src/schema/company_secrets.ts:6-10`).

## 3.

**위치**  
`docs/report/02-data-model.md:34`

**현재 본문 인용**  
> `requested_by_agent_id?` ⊕ `requested_by_user_id?`

**문제 카테고리**  
정합성

**제안 수정안**  
> `requested_by_agent_id?`, `requested_by_user_id?`

설명도 필요하면:

> 요청자는 agent/user 둘 다 nullable이며, 생성 라우트는 board user가 특정 requesting agent를 대리해 approval을 만들 때 두 값을 함께 채울 수 있다.

**근거**  
`approvals` 스키마에는 두 컬럼이 nullable로만 정의되어 있고 XOR 제약이 없다(`packages/db/src/schema/approvals.ts:9-17`). 생성 라우트도 board actor의 `requestedByUserId` 를 채우면서, 요청 본문에 `requestedByAgentId` 가 있으면 이를 함께 저장할 수 있다(`server/src/routes/approvals.ts:89-96`). 따라서 `⊕` 표기는 현재 구현보다 강한 불변조건을 암시한다.

## 8.

**위치**  
`docs/report/02-data-model.md:116`

**현재 본문 인용**  
> 이 패키지는 `package.json` 에서 `embedded-postgres@18.1.0-beta.16` 으로 핀(pin)되어 있고... PostgreSQL 17 (full native) 을 그대로 호스트하므로...

**문제 카테고리**  
정합성

**제안 수정안**  
> 이 패키지는 `packages/db/package.json` 에서 `embedded-postgres@^18.1.0-beta.16` 으로 선언되고, lockfile과 patch 항목에서 `18.1.0-beta.16` 으로 고정되어 있다. WASM이 아니라 플랫폼별 native PostgreSQL 바이너리를 띄우므로 PGlite보다 호환성이 좋다.

`PostgreSQL 17`이라고 특정하려면 실제 런타임 `SHOW server_version` 결과를 근거로 추가해야 한다.

**근거**  
`packages/db/package.json` 의 dependency는 exact pin이 아니라 caret 범위다(`packages/db/package.json:46-50`). exact 고정은 root patchedDependencies와 lockfile 쪽에 있다(`package.json:64-67`, `pnpm-lock.yaml:5080-5082`). 플랫폼별 optional binary도 `@embedded-postgres/*@18.1.0-beta.16` 으로 잠겨 있다(`pnpm-lock.yaml:1601-1644`). 현재 근거만으로 "PostgreSQL 17"을 특정하는 것은 맞지 않고, 문서의 research note도 PostgreSQL 14~18 버전을 npm 태그로 선택한다고 설명한다(`docs/research/06-drizzle-pglite.md:32`).
