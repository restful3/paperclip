R04는 5개 CLI의 큰 분류와 Paperclip 로컬 어댑터 연결 구조를 대체로 잘 잡고 있습니다. 다만 Cursor 생태계 세부 주장 일부가 인용한 블로그와 맞지 않고, Paperclip §4의 어댑터 공통 동작 설명이 실제 코드보다 넓게 일반화되어 수정이 필요합니다.

### 1. Cursor CLI 출시·키 범위 세부 주장 근거 불일치

**위치**: L33

**현재**: `1월 2026 이후 동일 토큰으로 IDE Agent와 헤드리스 CLI 둘 다 호출되며, 별도의 Background Agents API 도 같은 키로 묶여 있다(단 Free Plan 키는 헤드리스 CLI만 허용)([Cursor blog](https://cursor.com/blog/cli)).`

**문제**: [ACCEPT] 사실/인용 완결성 문제. 인용한 Cursor CLI 블로그는 2025-08-07 글이며 CLI/headless 사용 가능, 구독에 포함된 모델 사용, terminal/remote에서 여러 agent 실행 가능 정도를 설명합니다. 현재 문장의 “1월 2026 이후”, “Background Agents API도 같은 키”, “Free Plan 키는 headless CLI만 허용”은 해당 URL에서 확인되지 않습니다. 출처: https://cursor.com/blog/cli

**제안**: 이 문장은 Cursor 블로그가 직접 뒷받침하는 범위로 낮춰 “Cursor Agent CLI는 Cursor 구독의 일부로 editor 밖 CLI/headless 환경에서 같은 Cursor Agent를 사용할 수 있게 한다” 정도로 정리하세요. Background Agents API/Free Plan 제한을 유지하려면 그 내용을 직접 말하는 별도 Cursor 공식 출처를 붙이세요.

### 2. OpenCode build/plan 근거 출처 부족

**위치**: L41

**현재**: `헤드리스 자동화는 `opencode run "<prompt>" --format json`이 표준이고 "build"(쓰기 가능) / "plan"(읽기 전용) 두 빌트인 에이전트가 분리돼 있다.`

**문제**: [ACCEPT] 인용 완결성 문제. `opencode run --format json`, `serve`, `web` 같은 CLI 표면은 현재 붙은 OpenCode CLI 문서로 확인됩니다. 하지만 build/plan 두 기본 에이전트 설명은 CLI 문서가 아니라 OpenCode Agents/Config 문서의 내용입니다. 현재 문장 끝의 출처가 `[OpenCode MCP]`까지 붙어 있어, build/plan의 직접 근거가 불명확합니다. 출처: https://opencode.ai/docs/cli/ 및 https://opencode.ai/docs/agents/

**제안**: build/plan 문장 뒤에 OpenCode Agents 문서를 추가 인용하세요. 예: “OpenCode는 Build와 Plan 두 primary agent를 기본 제공한다([OpenCode Agents](https://opencode.ai/docs/agents/)).”

### 3. Paperclip 어댑터의 tool call 추출 일반화

**위치**: L51

**현재**: `JSONL stdout을 파싱해 token usage·툴 호출·세션 ID를 추출해 다시 control plane에 보고한다.`

**문제**: [ACCEPT] 사실 정확성 문제. 공통 반환 표면은 token usage, session id, cost/summary에 가깝고, tool call을 모든 어댑터가 구조화해 보고하지는 않습니다. 예를 들어 `cursor-local` 파서는 `usage`, `sessionId`, `costUsd`, `summary`, `errorMessage`를 반환하지만 tool call 목록은 반환하지 않습니다(`packages/adapters/cursor-local/src/server/parse.ts:52-149`). `codex-local`도 `sessionId`, `summary`, `usage`, `errorMessage` 중심입니다(`packages/adapters/codex-local/src/server/parse.ts:14-70`). `opencode-local`은 tool call 자체가 아니라 `toolErrors`를 분리합니다(`packages/adapters/opencode-local/src/server/parse.ts:22-87`). Pi는 tool call 목록을 파싱하지만(`packages/adapters/pi-local/src/server/parse.ts:14-31`, `:166-190`), 실행 결과 공통 반환에서는 usage/session/cost가 중심입니다(`packages/adapters/pi-local/src/server/execute.ts:749-761`).

**제안**: “JSONL stdout을 파싱해 token usage·cost·세션 ID·요약/오류를 회수하고, 도구별로 가능한 경우 tool 이벤트나 tool error를 보조 정보로 처리한다”처럼 공통 보장과 일부 어댑터별 보조 파싱을 분리하세요.

### 4. Pi 어댑터 인자 배열에서 최종 프롬프트 누락

**위치**: L51

**현재**: ``pi-local/src/server/execute.ts`는 `["--mode", "json", "-p", "--append-system-prompt", ..., "--provider", p, "--model", m, "--thinking", t, "--tools", "read,bash,edit,write,grep,find,ls", "--session", sessionFile, "--skill", skillsDir]`를 그대로 조립한다.`

**문제**: [ACCEPT] 사실 정확성 문제. 실제 `buildArgs`는 이 플래그 뒤에 `extraArgs`를 붙인 다음, 사용자 프롬프트를 마지막 positional argument로 추가합니다(`packages/adapters/pi-local/src/server/execute.ts:618-641`). 또한 `--provider`, `--model`, `--thinking`은 각각 값이 있을 때만 들어가는 조건부 인자입니다(`packages/adapters/pi-local/src/server/execute.ts:628-630`). 사용자가 특히 인자 ordering 검증을 요청했으므로 최종 프롬프트 누락은 substantive합니다.

**제안**: Pi 배열을 “`--mode json`, `-p`, `--append-system-prompt`, 조건부 `--provider/--model/--thinking`, `--tools`, `--session`, `--skill`, `extraArgs`, 마지막 `userPrompt` 순서”로 정정하세요.
